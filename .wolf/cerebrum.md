## Key Learnings

- **Behavior authoring has THREE layers — keep them in sync.** A new card behavior must be wired in up to three places, and they are independent: (1) **engine** (SQL migration) — makes it work; (2) **Zod schema** [card-behavior-schema.ts] — validates hand-written/AI JSON; (3) **guided builder** [card-behavior-registry.ts effect entries + card-behavior-builder.ts BuilderForm for top-level props + CardBehaviorForm.tsx for the widget] — exposes it in the no-code form. The guided form is driven by the **registry** (`effectsForContext`), NOT the Zod schema, so adding to the schema does NOT surface it in the form. Effect-level options (e.g. create_token `tapped`, new contexts) = one registry entry. Top-level card props (flashback, loyalty_abilities) are NOT in the registry — they need a `BuilderForm` field + build/parse wiring + a widget. If a script uses anything the form can't model, `parseScriptToForm` returns null and the editor drops to JSON mode. The cheap mistake (made this session): updating engine + schema + LLM docs but forgetting the registry, so the card only authors via AI/JSON, not the guided form.
  - **Three distinct wiring SHAPES in the guided builder, depending on what the behavior IS:** (a) a **resolution effect** (deal_damage, create_token, …) applied at trigger/spell/rider resolution = ONE registry entry (`effectsForContext` drives the dropdown + fields). (b) a **flat top-level prop** (flashback string) = a `BuilderForm` scalar field + build/parse + one widget. (c) a **`continuous_effects` entry** (keywords, static anthems/lords) — these are STATIC, registered at rebuild, NOT resolution effects, so they bypass the registry entirely: they need a dedicated `BuilderForm` list + a hand-written parse that partitions `continuous_effects` (`parseContinuousEffects`) + a bespoke editor component. Static type-lords ("Other Zombies you control get +1/+1" = `{type:'pump', affected:'controller', payload:{power,toughness,creature_type,exclude_source}}`) are shape (c): wired via `BuilderForm.staticBuffs` + `StaticBuffEditor` in CardBehaviorForm.tsx (Cemetery Reaper, this session). Engine already existed (typed-lords mig 164 = `Zombie Lord Test`); only the form layer was missing.
  - **`BuilderStaticBuff` gained a `scope` ('controller' | 'all') field** so the form authors both "creatures you control" (affected:'controller') and global/sliver anthems "all creatures" (affected:'all', engine support from mig 145). `parseStaticBuff` accepts either `affected` value.
- **A type in the Zod schema does NOT prove engine support (bug-464).** The schema (`card-behavior-schema.ts`) is permissive — it validates author *intent* and lists capabilities the engine may not implement (e.g. `sacrifice_self` cost was a known cost type but `activate_ability` raised on it). When triaging "is X supported," grep the migrations for the actual handler; the engine is the source of truth for what runs. **`sacrifice_self` activated-ability cost (mig 175):** reproduced `activate_ability` (from mig 162) with a `v_has_sac` branch that `put_in_graveyard`s the source after other costs, before the ability hits the stack (`put_action_on_stack` tolerates a non-battlefield source). Form: the `effect`-kind activated ability gained a `sacSelf` boolean (a "Sacrifice this" checkbox); mana abilities reject `sacSelf` on parse. Commander's Sphere now works (engine + form).
- **"Exile a creature card from a graveyard" activated-ability cost + create_token effect (mig 178, Cemetery Reaper).** Two activated-ability gaps closed at once: (1) a new `exile_from_graveyard` cost (`{type:'exile_from_graveyard', type_line:'creature'}`) — the chosen graveyard card is passed as `p_target_card_id` (these abilities' EFFECT is untargeted, so the target slot carries the cost target); the engine validates it's a matching card in ANY graveyard and exiles it (controller:=owner). (2) `create_token` as an activated-ability effect — `activate_ability` had NO token builder and `put_action_on_stack` has no token handler, so it routes the token through a DIRECT INSERT of a `spell_effect` stack item (reusing the spell-effect resolver → apply_triggered_ability_effects' create_token, incl. tapped). Form: an "Exile a creature from a graveyard" checkbox on the effect-kind activated ability (creature filter only; other filters → JSON mode). **Token-name two-list gotcha:** the form's `EFFECT_TOKEN_NAMES` (registry) and the SEEDED test-card tokens are DIFFERENT lists — a token must be in BOTH to be form-authorable AND engine-resolvable in tests. `Zombie Token` is in EFFECT_TOKEN_NAMES but wasn't seeded; added it to test-cards.json. The in-game PICKER is done: ControllerListV4 detects an `exile_from_graveyard` cost (`needsGraveyardCard`), and the activated-ability target picker (`abilityPick.canTargetGraveyard`) lists creature cards from ANY graveyard (`graveyardCreatures = boardCards.filter(zone==='graveyard' && creature)`) — choosing one calls activate_ability with it as targetCardId. Also taught getAbilityEffect/ABILITY_EFFECT_TYPES that `create_token` is a supported untargeted ability effect. Committed via hunk-split (the ControllerListV4 two-column reflow stays uncommitted). Cemetery Reaper ability #2 is now fully playable end-to-end.
- **Watcher-trigger FILTER in the guided form (Champion of the Perished).** The form's triggers were event+effects only; the watcher events (creature_entered/died/got_counter) need a `filter` (`{type_line, controller, exclude_self}`) to be useful, which wasn't exposed (engine mig 165 + schema already supported it). Added `BuilderTrigger.filter` (`{typeLine, controller, excludeSelf}`) + `parseTriggerFilter` + a filter UI shown only for watcher events (a type box, a you/opponent/anyone select, an "another (not this)" checkbox). **Controller-default gotcha:** the ENGINE defaults an ABSENT `controller` to `'you'` (mig 165 line 76: `coalesce(... ,'you')`), and `'any'` needs to be EXPLICIT (`else true` branch). So the form default is `'you'`, build OMITS controller when `'you'` (canonical minimal JSON) and emits it for opponent/any. A bare watcher (no filter) already means "a creature you control". Champion of the Perished = `creature_entered` + `{type_line:'Zombie', exclude_self:true}` (controller you implied) + `add_counters 1`.
- **Mass typed P/T debuff `pump_all` (mig 179, Crippling Fear).** "Choose a creature type. Each creature that isn't of the chosen type gets -3/-3 until end of turn." Three pieces: (1) **`exclude_type` in the P/T fold** — `card_layered_power/toughness` (reproduced from mig 164) gained a `payload.exclude_type` flag that INVERTS the creature_type filter (creatures NOT of the type get the pump). (2) **`apply_mass_pump_until_eot(session, source, controller, effect)`** helper — inserts a `pump` continuous effect (`affected:'all'` when scope!='controller'; affected_player_id=null) with `expires_at_phase='ending'/step='cleanup'` (the existing sweep removes it), then runs `move_lethal_damaged_creatures_to_graveyard` (a -X/-X reads EFFECTIVE toughness, so it can kill). (3) **`choose_creature_type` (submit_decision, reproduced from mig 170) now injects the chosen type into a `pump_all`'s `creature_type`** and applies pump_all via the helper (separating it from the rest, which still go through apply_triggered_ability_effects). **Leaner choice:** handled pump_all inside choose_creature_type's branch (the injection HAS to live in submit_decision anyway) rather than ALSO reproducing the huge apply_triggered_ability_effects. **Form: now authorable.** Added `choose_creature_type` (effect-list, contexts:['spell'], itemContext:'spell') + `pump_all` (power/toughness/scope/creature_type/exclude_type, contexts:['trigger','spell']) as REGISTRY entries — that was sufficient (the form, dropdown, defaults, parse/build all derive from the registry; NO builder-union or component change needed, since BuilderSpellEffect is a loose cast and BUILDER_SPELL_EFFECT_TYPES = effectsForContext('spell')). Crippling Fear authoring: Spell effect → "Choose a creature type, then…" → inner "All creatures get ±X/±X" (power -3, toughness -3, scope all, ✓ "Creatures NOT of that type", Creature type BLANK — injected from the choice). Distant Melody's count-draw still isn't form-representable (count-amounts aren't), but choose_creature_type + pump_all is. Authoring JSON: `{type:'choose_creature_type', effects:[{type:'pump_all', power:-3, toughness:-3, scope:'all', exclude_type:true}]}`.
- **`choose_player` = "target player X" — the chosen player is the recipient of ALL inner effects.** On resolution the engine REWRITES every inner effect's `recipient` to `'controller'` and applies them with `p_controller_id` = the chosen player (`submit_decision`, mig 170 line ~582: `e.value || jsonb_build_object('recipient','controller')`). So inside `choose_player` the recipient field is meaningless (overridden), and the subject is the chosen player, never the caster. **"Target player draws N" (Deep Analysis) = `choose_player(filter:'any') → draw N`** — already fully supported (engine + form + flashback); no new primitive needed (another false-alarm triage item, like Champion of the Perished). **Form labeling caveat (fixed this session):** the `draw` effect was labelled "You draw cards", which misread inside choose_player. Fixed by (a) neutralizing draw's label to "Draw cards" (the drawer is the effect's controller — source's controller in a trigger/spell, caster in a `then` rider, chosen player in choose_player), and (b) binding the subject at the wrapper: choose_player's label is "Choose a player, then that player…" and its effects field label is "That player:". NOTE: the registry disambiguates effect VARIANTS by field-shape (`resolveEffectDef`), so you can't add a same-shape variant (e.g. a "chosen-context draw" with the same `[amount]` fields) — distinguish by a differing field (the rider variants drop `recipient`) or fix the label instead.
- **Flashback (mig 174):** This engine moves a non-permanent spell out of its cast zone AT CAST TIME (hand→graveyard inside `cast_spell_effect`), keeping a separate `spell_effect` stack item that resolves the program. So flashback's "then exile it" is modelled as the graveyard→exile move at cast time (card ends in exile whether the spell resolves or is countered — correct for flashback). `cast_spell_effect` already had NO source-zone filter, so a graveyard source was structurally castable; the flashback branch reads the card's `script ->> 'flashback'` cost, pays it instead of the printed cost, and requires it (graveyard cast without a flashback cost now raises). Tapped tokens: the `create_token` branch of `apply_triggered_ability_effects` honours a `tapped` flag (spell-effect tokens route there via apply_trigger_effects → apply_targeted_triggered_ability_effects → apply_triggered_ability_effects).
  - **Flashback "Pay N life" additional cost (mig 176):** flashback is a plain mana-cost STRING (`script.flashback`) consumed in 5 places (schema, card-behavior type/normalize, builder build/parse, `cast_spell_effect`, ControllerListV4 button). An additional life payment is a SEPARATE optional `flashback_life` integer (NOT an object — keeps every string consumer untouched). `cast_spell_effect`'s graveyard branch pays the mana then deducts `flashback_life` from `game_session_players.life_total` (raises if life < cost; can't pay life you don't have). Form: a "+ pay N life" number beside the flashback mana input; build emits `flashback` (possibly '') whenever flashback mana OR life is present so the engine's graveyard-cast gate (needs the `flashback` key) still fires for a life-only flashback. Deep Analysis = `{flashback:'{1}{U}', flashback_life:3}`. The controller flashback button now also renders "+N life".
  - **Flashback that does DIFFERENT/extra stuff than the hand cast (mig 177, the "Increasing" cycle).** Optional `flashback_effect: { actions: [...] }` (same shape as spell_effect) REPLACES spell_effect when cast from the graveyard. **Engine-ENFORCED, not client-trusted:** `cast_spell_effect` selects the program by cast zone (`if v_source_zone='graveyard' and flashback_effect.actions present → use it, else p_actions`) BEFORE the X-resolution — so the client can pass the normal actions and the engine still runs the flashback program (can't be substituted). Because the engine enforces it, NO client change was needed for untargeted "do more" cards (the controller keeps sending the normal actions). Authoring: `BuilderForm.flashbackEffect` (a second spell-action list, rendered by the factored `SpellActionList` shared with the normal Spell-effect section); build emits `flashback` (possibly '') whenever any flashback facet (mana/life/effect) is present. Increasing Devotion = `spell_effect` create 5 + `flashback_effect` create 10. Caveat: a flashback_effect that needs NEW targets/decisions the client didn't set up for the normal cast isn't wired in the controller yet (untargeted programs only).
- **Cast-from-graveyard (mig 173):** Alternate cast ZONES are modelled as a player-scoped permission, not a rewrite of the cast path. A `cast_from_graveyard` continuous-effect row (`affected_player_id` + `payload.type_line` filter, `expires_at_phase='ending'/step='cleanup'`, NOT flagged `registered_from_card_script` so rebuild leaves it) is written by a `grant_cast_from_graveyard` effect branch; `cast_card_from_hand` then accepts `zone in ('hand','graveyard')` and gates a graveyard source on a matching permission. End-of-turn sweep is automatic via the existing `expire_continuous_effects_for_step`. This was the deferred "Noctis-class" frontier — turned out small because the cast path is one function and the permission reuses existing continuous-effect machinery.

- **Triggered abilities (migrations 076–077):** events detected by DB row triggers (mirrors the `cease_token` pattern). `fire_zone_change_triggers` on `game_cards` handles ETB (zone→battlefield) and dies (battlefield→graveyard); `fire_upkeep_triggers` on `game_turn_state` step→upkeep; `fire_attack_triggers` on `game_combat_assignments` insert.
