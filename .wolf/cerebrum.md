## Key Learnings

- **Project:** LeylineSync
- **Description:** Realtime Magic: The Gathering style board/controller app built with Next.js and Supabase.
- **Card behavior resolution chain:** runtime reads behavior as `coalesce(game_cards.copied_script, cards.script)` across ~10 SQL functions (casting, rebuild_scripted_continuous_effects, activate_ability, combat keyword registration). `copied_script` = per-instance copy override; `cards.script` = base.
- **The Scryfall importer never writes `cards.script`** (scripts/import-scryfall-cards.mjs `mapScryfallCardToRow` omits it) and upserts `onConflict: 'id'`, so `ON CONFLICT DO UPDATE` leaves `cards.script` untouched → authored behavior survives reimports. This is why Phase 2 authored directly onto `cards.script` instead of building a separate oracle_id-keyed override table.
- **`cards.id` is the Scryfall printing id, not oracle identity.** `oracle_id` (added migration 075) is the card identity shared across printings; importer now persists it. Use `relink_card_scripts()` to re-attach a script if the representative printing for an oracle changes between imports.
- **State-based actions:** 0-toughness death (rule 704.5f) ignores indestructible; lethal marked damage (704.5g) is prevented by indestructible. Both handled in `move_lethal_damaged_creatures_to_graveyard` (migration 074).
- **Triggered abilities (migrations 076–077):** events detected by DB row triggers (mirrors the `cease_token` pattern). `fire_zone_change_triggers` on `game_cards` handles ETB (zone→battlefield) and dies (battlefield→graveyard); `fire_upkeep_triggers` on `game_turn_state` step→upkeep; `fire_attack_triggers` on `game_combat_assignments` insert. Each match calls `fire_card_triggers` → `enqueue_triggered_ability`, which inserts a `triggered_ability` stack item with effects + controller + source card name (`label`) baked into the payload (so resolution + display survive the source leaving). `resolve_top_of_stack` applies untargeted effects (gain_life/lose_life/deal_damage to controller|each_opponent, draw). Zone-change trigger name `trg_a_fire_zone_change` sorts before `trg_cease_token_off_battlefield` so a token's dies trigger enqueues before cease-to-exist; `game_stack_items.source_card_id` is ON DELETE SET NULL so the stack item survives, and `get_stack_items` coalesces `source_card.name` with the payload `label`. **Only targeted triggers remain unbuilt.**
- **`get_stack_items` was redefined in migrations 030→032→033**; the live version (033) has extra `target_player_id`/`target_username` columns. When reproducing it, base on 033, not 030, or you silently drop columns the client `StackItem` type depends on.
- **Triggered-ability effects live in `apply_triggered_ability_effects` (migration 078)** — a helper `resolve_top_of_stack` calls for the `triggered_ability` branch. To add a new auto-resolved trigger effect, edit this one function (no longer reproduce the whole resolve_top_of_stack). Supported: gain_life, lose_life, deal_damage (recipient controller|each_opponent), draw, create_token (by token name from seeded catalog), add_counters (+1/+1 on source). create_token inserts battlefield game_cards which itself fires the ETB zone-change trigger — intended.
- **Card behavior authoring stack:** `/cards/behavior` editor has Form mode (guided builder, `components/CardBehaviorForm.tsx` + model/codec in `lib/game/card-behavior-builder.ts`), JSON mode, and ✨ Generate-with-AI. The builder's vocabulary constants are the single source of truth: the form dropdowns, the LLM prompt (`lib/game/card-behavior-llm.ts` → `buildBehaviorAuthoringGuide`), and validation (`validateCardScript`) all derive from / agree on them. To add a new effect type, extend the builder + the Zod schema; the form and AI follow. AI route: `app/api/cards/generate-behavior/route.ts` — auth-gated, `claude-opus-4-8`, adaptive thinking, cached system prompt, validates output with `validateCardScript` and retries once; guarded by `ANTHROPIC_API_KEY` (returns 501 if unset). SDK: `@anthropic-ai/sdk`.
- **Reusable internal effect helpers:** `draw_card`/`adjust_player_life` RPCs gate on `auth.uid()` so they can't be reused for an arbitrary trigger controller; triggered-ability resolution applies life/draw inline against the payload's `controller_player_id` instead.
- **Spell effect types via the stack (migration 079):** `spell_effect.actions` types map to stack action types in `getSpellPlan` (ControllerListV4): `deal_damage`→deal_damage_creature/player, `pump`→pump_creature, `destroy/bounce/tap/untap`→`*_creature` (kind `creature_effect`, one shared V4 picker via `CREATURE_EFFECT_MAP`), `draw`→`draw_cards` (untargeted, kind `draw`, cast button calls onDrawCards not onCastCard). `draw_cards` resolution reuses `apply_triggered_ability_effects` with a synthetic `{type:draw}` effect (pass null source). destroy→owner's graveyard (fires `dies` triggers, intended), bounce→owner's hand (does NOT fire dies — that's only battlefield→graveyard); both reset is_tapped/damage/counters and controller→owner. **Gotcha:** `doesCardRequireStackTarget` returns true for ANY instant/sorcery whose oracle_text contains "target", which would gate these on a non-empty stack (counterspell behavior). `getSpellPlan` must classify the new kinds BEFORE the counterspell fallback, and `canCastHandSpell` special-cases damage/pump/creature_effect/draw to cast at natural speed bypassing `getCanQuickCast`.
- **Trigger event detection is fully decoupled from targeting (migrations 076–082).** `fire_card_triggers(session, game_card, events[])` scans the card's effective script for `triggered_abilities` whose `event` matches and calls `enqueue_triggered_ability` — identically for every event. So adding a new event = add a DETECTOR that calls fire_card_triggers with new event names; targeting/target_controller work automatically. Detectors: `fire_zone_change_triggers` (game_cards zone change → enters_the_battlefield/dies/leaves_the_battlefield), `fire_turn_step_triggers` (game_turn_state step change → beginning_of_upkeep/draw_step/end_step; migration 082 replaced the upkeep-only `fire_upkeep_triggers`), `fire_attack_triggers` (game_combat_assignments insert → attacks), `fire_block_triggers` (game_combat_blockers insert → blocks), `fire_target_triggers` (game_stack_items insert for creature-target action_types → becomes_targeted; INSERT-only so it's loop-free and lands above the targeting item). Builder vocabulary `BUILDER_TRIGGER_EVENTS` (card-behavior-builder.ts) must list a new event's primary value or the form rejects it (parseTriggeredAbilities) and the LLM guide won't mention it; SQL accepts the aliases. Event field in Zod schema is just z.string() (no validation gate).
- **Targeted creature triggers + controller restriction (migrations 080–081):** triggered_ability stack items carry `target_required`/`target_card_id`/`target_controller` in payload. Controller picks the creature via `choose_triggered_ability_creature_target` RPC (client: `chooseTriggeredAbilityCreatureTarget`; UI: `TargetedTriggerPrompt` + `mustChooseTriggerTarget` blocks passing priority via `blockPassReason`). A trigger effect requires a chosen creature target ONLY when its type is targetable (deal_damage/destroy/bounce/tap/untap/add_counters) AND `target_type` is **creature-only** (`behavior_target_type_is_creature_only`) — "any target" damage triggers fall through to `apply_triggered_ability_effects`' each_opponent default instead of being dropped. `target_controller` (any|opponent|you; normalized from any/opponent/you/self/controller) enforced by SQL helpers `session_has_targetable_creature` + `creature_target_controller_ok`, used in enqueue (existence check), choose, resolve fizzle guard, and put_action_on_stack (spell casts). Client mirror: `creatureMatchesController` / `readTargetController` filter pickers; SpellPlan kinds carry `targetController`; put-on-stack action fns take a trailing optional `targetController` param. No-legal-target softlock avoided: enqueue won't enqueue if no legal creature; resolve guard only raises when a legal creature still exists, else fizzles.
- **Battlefield→graveyard chokepoint `put_in_graveyard(session, game_card)` (migration 084, Phase 0 of the rules-engine cleanup).** Single primitive for the dies/leaves-to-grave transition (returns bool; standard cleanup: zone→graveyard at owner's next position, controller→owner, is_tapped=false, damage_marked=0, dealt_deathtouch_damage=false, plus_one_counters=0). It does NOT rebuild continuous effects or fire triggers — the per-row zone-change trigger fires dies/leaves automatically; callers rebuild once after a batch. All three death sites route through it: `move_lethal_damaged_creatures_to_graveyard` (now snapshots the dying id set with array_agg, then loops put_in_graveyard — preserves "evaluate dying set once"), `resolve_top_of_stack` destroy_creature, `apply_targeted_triggered_ability_effects` destroy. This is the seam for the future finality counter / death-replacement layer (change one function) and for moving resolution to app code. Note: exile→exile and bounce→hand stay inline (different zones); counter_spell stack→graveyard is NOT a battlefield death and is untouched. Deliberate harmless unification: SBA deaths now also reset controller→owner (graveyard cards are owned, not controlled). **The decision (from the architecture discussion): keep authority + atomicity in Postgres; centralize in SQL (put_in_graveyard, future apply_effect / effective_characteristics); only build a pure-TS `reduce(state,action)` rules core at the replacement-effect/layer frontier (Noctis-class) — justified by shared client/server optimistic UI + deterministic replay, NOT a rewrite-now.**
- **Effect roadmap + exile/mill (migration 083, README "Effect roadmap" `<details>` block).** Adding a creature-targeting effect = mirror destroy everywhere it appears: the action_type CHECK constraint, `trigger_effect_requires_creature_target` list, `apply_targeted_triggered_ability_effects` branch, `put_action_on_stack` (not-in list + combined creature-target validation branch + payload case), `resolve_top_of_stack` branch, plus client `CREATURE_EFFECT_MAP` (key = spell action type e.g. `exile`→`exile_creature`), `TargetedCreatureActionType`, and schema (`z.enum(['destroy','exile','bounce','tap','untap'])` + KNOWN_V2_ACTION_TYPES). exile → owner's exile zone (fires leaves_the_battlefield, not dies). Adding an auto-resolved (untargeted) effect = edit ONLY `apply_triggered_ability_effects` + Zod member + KNOWN_V2_ACTION_TYPES + LLM prose; mill lives there (recipient controller default/each_opponent/each_player, mills top-of-library → graveyard). mill is trigger-only — no spell action yet (player-targeted mill needs a player picker). Effects authored via JSON/AI + Zod, NOT the guided form (BUILDER_EFFECT_TYPES), same as the targeted-creature effects — only add to BUILDER_EFFECT_TYPES if you also add form model/parse/serialize/UI.
- **Effective-script accessor `effective_script(session, game_card)` (migration 085, Phase 0).** Single STABLE SECURITY DEFINER function returning `coalesce(game_cards.copied_script, cards.script)` (null if the card isn't in the session). Replaces the inlined coalesce in the 3 *live* callers: `register_card_continuous_effects` (was 067), `activate_ability` (was 072), `fire_card_triggers` (was 077) — all other historical hits (044/048/049/050/055/063/064/067 register redefinitions) were superseded. Each caller reproduced verbatim with ONLY the script fetch swapped (activate_ability keeps its zone+ownership check by selecting `zone` alone then calling the accessor; register keeps reading `cards.keywords` separately since keywords are base-card data, not script). Behavior-preserving today; it is the one seam where DFC active-face → face-specific script selection will live. NOTE: `rebuild_scripted_continuous_effects` (063) uses a *different* shape — `card_behavior_has_continuous_effects(coalesce(copied_script,'{}'))` (presence prefilter, no fallback to cards.script) — left untouched, not part of this accessor.
- **Creature-effect primitive `apply_creature_effect(session, kind, target_card_id, params)` (migration 086, Phase 0).** Single SECURITY DEFINER function owning every targeted-creature mutation — kind ∈ {deal_damage, destroy, exile, bounce, tap, untap, add_counters, pump}; params jsonb carries `amount` (deal_damage/add_counters) or `power`/`toughness` (pump). No-ops if target null or off-battlefield (fizzle). destroy → put_in_graveyard; deal_damage re-runs move_lethal SBAs. Both former call sites now delegate: `apply_targeted_triggered_ability_effects` passes the effect `type` as the kind (non-creature effects still defer to `apply_triggered_ability_effects`); `resolve_top_of_stack` collapsed its 8 `*_creature` action branches into ONE `in (...)` branch that derives the kind via `regexp_replace(action_type,'_creature$','')` and hands the whole payload as params. This was the last logic duplication of the "mirror destroy in 2 places" tax. **Updated rule for adding a creature-targeting effect:** implement the mutation ONCE in apply_creature_effect, then only the declarations/validation remain (action_type CHECK constraint, `trigger_effect_requires_creature_target` list, `put_action_on_stack`, client `CREATURE_EFFECT_MAP`/`TargetedCreatureActionType`, Zod enum + KNOWN_V2_ACTION_TYPES) — no duplicated branch logic. Safe because `trigger_effect_requires_creature_target` only admits types apply_creature_effect handles, so a trigger effect can't hit its `else raise`. With this + effective_script (085) + put_in_graveyard (084), the two SQL Phase-0 centralizations from the Decision Log are complete; remaining Phase-0/1 work is the untargeted side already lives solely in apply_triggered_ability_effects, then the generic "pending decision" state machine.
- **Rules-engine test harness (`tests/`, the "test chamber").** Programmatic regression suite driving the real RPCs against **local Supabase** Postgres. Key facts: (1) uses a **direct `pg` connection, not supabase-js**, because `auth.uid()` resolves from `request.jwt.claims ->> 'sub'` and only a raw connection can set that GUC on the same connection before the RPC — `harness/db.ts` `asPlayer(client, playerId, fn)` does `set_config('request.jwt.claims', {sub,role}, true)` + `set local role authenticated`, then `reset role` so assertions read tables as postgres (RLS bypassed). (2) Each test runs in a transaction rolled back at the end (`withRolledBackTx`) → isolation without `db reset`. (3) **No `auth.users` seeding needed** — `game_session_players.player_id` has NO FK to auth.users (migration 009), so any random uuid is a valid player. (4) Setup reuses existing `dev_*` RPCs (`dev_spawn_card`, `dev_set_turn_state`, `dev_clear_summoning_sickness`). (5) `resolve_top_of_stack` can be called directly (doesn't require priority to have passed — that's `pass_priority`'s job), so it's a clean unit-resolve. Runner: `node --import tsx --test` (node:test, no vitest); Node's `--test` glob is unreliable on Windows so package.json points at the file explicitly. `Scenario` API in `harness/scenario.ts`: create/spawn/spawnCreature/setTurn/as(seat).putOnStack|activate/resolveStack + inspectors. **VERIFIED WORKING 2026-06-02** — proving trio (R1 destroy→put_in_graveyard, R8 activate→effective_script, R11 draw) passes. **Local schema = SQUASH, not migration replay** (see Decision Log + bug-104): `supabase/migrations/` holds only `00000000000000_baseline.sql` (full hosted `db dump --schema public`, reproduces prod incl. 079-086) + `00000000000001_local_test_relax_fks.sql` (LOCAL ONLY — drops the auth.users/profiles FK chain so throwaway player UUIDs work; never apply to hosted). The 88 incremental migrations were moved to `supabase/migrations_archive/` (Supabase ignores them; kept as history). `% Test` cards are schema-only-dump-excluded, so the harness seeds them from `tests/fixtures/test-cards.json` via `ensureTestCards` (committed once, idempotent). Gotchas burned in: turn **phases** are `beginning|main_1|combat|main_2|ending` (NOT the step names — `precombat_main` is a step; a main-phase instant window = phase `main_1` + step `precombat_main`); `asPlayer` cleanup must be try/catch-wrapped or a failed RPC's real error is masked by 25P02 on the aborted tx; FK chain is `game_cards.owner_id -> profiles.id -> auth.users.id`. `supabase/_hosted_schema.sql` is the gitignored scratch dump used to author the baseline.
- **Validator catch-all gotcha (card-behavior-schema.ts):** `UnknownV2ActionSchema` accepts any action type NOT in `KNOWN_V2_ACTION_TYPES`. Adding a type name to that list WITHOUT also adding an explicit union member makes the catch-all reject it → validation fails. Always add both together (learned adding pump/destroy/bounce/tap/untap).

- [2026-05-27] **MTG combat — multiple blockers damage order:** When multiple creatures block a single attacker, it is the **ATTACKER** (not the defender) who assigns damage order among blockers. The defender's declare blockers phase only determines *which* creatures block *which* attacker. Do not show a "stel schade-volgorde in" or damage-ordering UI element on the defender's declare blockers screen.

## Decision Log

<!-- Significant technical decisions with rationale. Why X was chosen over Y. -->

- [2026-06-02] **Local DB = squashed dump baseline, not migration replay.** Discovered the incremental migration chain is not replayable from scratch (base tables cards/game_cards/game_players/profiles created out-of-band on hosted → bug-104; AND multiple returns-table functions change shape via create-or-replace → 42P13; AND Supabase's leading-version-digit keying makes mid-chain drop-shims impossible). Since hosted is applied MANUALLY (migrations/ isn't the prod driver), chose to squash for local: `supabase/migrations/` = one full-schema dump baseline + a local-only relax-FK migration; the 88 incrementals archived in `supabase/migrations_archive/` as history. New migrations still go in migrations/ and get pasted to hosted manually as before. Rejected: surgical per-table baseline (chain still didn't replay), drop-shim injection (version-PK collisions), out-of-band load (bespoke, db reset wouldn't work).
- [2026-06-02] **Test chamber = programmatic harness first (Option A), against local Supabase.** Chose a code-driven regression suite (drives real RPCs, asserts state) over an interactive in-app sandbox (Option B) as the first build — higher leverage (CI-able, protects the 084–086 refactors, encodes docs/test-plan-079-086.md permanently), and its scenario helpers (`place`/`cast`/`resolveStack`/`snapshot`) are the foundation a sandbox would later reuse, so it isn't throwaway. DB target: **local Supabase via Docker** over bare-Postgres-with-auth-shims or a hosted test project — the project is already supabase-initialized (config.toml) and Docker is installed; full prod fidelity (`db reset` replays the exact migrations), fast, offline. Cost accepted: tests need Docker Desktop running. Implementation keystone: direct `pg` + `request.jwt.claims` to satisfy `auth.uid()` (see Key Learnings). First slice = proving trio (R1/R8/R11) to validate the pipeline before expanding.
- [2026-06-01] **Rules engine stays in Postgres; refactor over rewrite.** Considered moving rules resolution to TypeScript (app code) vs keeping plpgsql. Chose: keep authority + atomicity + anti-cheat in Postgres (free transactional locking via `for update`, RLS, realtime), and instead pay down the duplication by centralizing in SQL — Phase 0 `put_in_graveyard` (done, migration 084), then `apply_effect` (one effect switch shared by spells/abilities/triggers) and `effective_characteristics` (one face/script accessor, also unlocks DFCs like Jill // Shiva). A pure-TS `reduce(state, action)` rules core is deferred to the **replacement-effect / layer-system frontier** (Noctis-class: "cast from graveyard" static permission, additional costs, finality counter = "if it would die, exile instead"), where SQL genuinely hurts. Rationale for eventually going TS there: shared client/server code → optimistic UI, and deterministic replay/undo for judge tooling. Rejected a full rewrite now: high risk for a solo project, and centralization makes either future cheap. Staging: Phase 0 centralizations → Phase 1 generic "pending decision" state machine (modal/scry/random, extends the existing `choose_*` pattern) → Phase 2 TS core only when replacement/layers arrive.
